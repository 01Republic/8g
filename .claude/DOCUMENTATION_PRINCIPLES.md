## 문서화 철학

8G 프로젝트의 모든 문서는 **실용성, 접근성, 유지보수성**을 최우선 가치로 합니다. 문서는 코드만큼 중요하며, 팀원들이 신속하게 정보를 찾고 이해할 수 있어야 합니다.

---

## 핵심 원칙

### 원칙 1: 한글 작성 원칙

**규칙**: CLAUDE.md 원문을 제외한 모든 문서는 **한글로 작성**합니다.

**이유**:

- 팀원 대부분이 한국어 모국어 사용자
- 기술 용어는 영문 병기 가능하지만 설명은 한글로
- 빠른 이해와 효율적인 커뮤니케이션

**적용 범위**:

- ✅ PRD.md - 한글
- ✅ product-specs/ 모든 문서 - 한글
- ✅ 기능 명세서 - 한글
- ✅ API 문서 - 한글
- ✅ 가이드 문서 - 한글
- ❌ CLAUDE.md - 영문 유지 (Claude Code 호환성)
- ❌ README.md - 영문 유지 (오픈소스 표준)

---

### 원칙 2: 간소화 원칙

**규칙**: 모든 문서는 **200줄을 초과하지 않습니다**.

**이유**:

- 긴 문서는 읽기 어렵고 유지보수가 힘듦
- 특정 정보를 찾기 어려움
- 문서 갱신 시 부담이 커짐

**200줄 초과 시 대응 방법**:

1. **섹션 분리**: 큰 섹션을 독립 문서로 분리
2. **하위 문서 생성**: 상세 내용은 별도 파일로
3. **링크 연결**: 상위 문서에서 하위 문서로 연결

**예시 구조**:

```
product-specs/
├── F-005-루프-설정.md (190줄) ← 개요 + 인덱스
└── F-005/                      ← 하위폴더
    ├── loop-execution.md (180줄)
    ├── loop-examples.md (200줄)
    └── loop-advanced.md (160줄)
```

---

### 원칙 3: 즉시 문서화 원칙

**규칙**: 개발 완료 직후 **반드시 문서를 최신화**하거나 최신화를 제안합니다.

**이유**:

- 개발 직후가 가장 정확한 기억을 가진 시점
- 나중으로 미루면 문서화가 누락됨
- 다른 팀원이 즉시 활용 가능

**적용 시점**:

| 개발 작업                | 문서화 대상                | 타이밍               |
| ------------------------ | -------------------------- | -------------------- |
| 새 기능 추가             | Feature Spec 생성          | PR 제출 전           |
| 기존 기능 수정           | Feature Spec 업데이트      | PR 제출 전           |
| API 변경                 | API 문서 업데이트          | 배포 전              |
| 아키텍처 변경            | Architecture 문서 업데이트 | 설계 완료 후 즉시    |
| 버그 수정                | Known Issues 업데이트      | 수정 완료 후         |
| 데이터베이스 스키마 변경 | DB 스키마 문서 업데이트    | 마이그레이션 적용 전 |

## 문서 분할 원칙 (200줄 초과 시)

### 자동 분할 규칙

**트리거**: 문서가 200줄 초과 시

**실행 절차**:

1. 문서와 동일한 이름의 하위폴더 생성
2. 메인 문서는 개요 + 링크만 유지 (180줄 이하)
3. 상세 내용을 논리적 단위로 분리하여 하위문서 작성 (각 200줄 이하)
4. 모든 하위문서 링크를 메인 문서에 추가

**폴더 명명 규칙**:

```
원본 파일: F-005-루프-설정.md
하위 폴더: F-005/

원본 파일: 데이터-모델.md
하위 폴더: 데이터-모델/
```

**메인 문서 구조**:

```markdown
# F-XXX: 기능명

## 개요

[핵심 개념 설명 - 20-30줄]

## 주요 컴포넌트

[간단한 구조 설명 - 30-50줄]

## 상세 문서

- **[실행 로직](./F-XXX/execution.md)** - 상세 실행 메커니즘
- **[사용 예시](./F-XXX/examples.md)** - 실제 코드 예시
- **[고급 기능](./F-XXX/advanced.md)** - 최적화 및 고급 활용

## 관련 문서

[링크들...]
```

**주기적 검사**:

- 개발 완료 후 문서 라인 수 확인
- 200줄 초과 시 즉시 분할
- CI/CD 자동 검사 권장 (향후 구현)

## 예외 처리

### 200줄 초과 허용 케이스

다음의 경우 **최대 300줄**까지 허용:

1. **종합 인덱스 문서** (product-specs/README.md 등)
2. **마이그레이션 가이드** (단계별 설명 필요)

- 섹션별로 앵커 링크 제공
- 가능한 경우 하위 문서 분리 권장
