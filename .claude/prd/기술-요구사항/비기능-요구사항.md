# 비기능 요구사항 (Non-Functional Requirements)

**문서 버전**: 1.0
**최종 수정일**: 2025-11-11
**상위 문서**: [기술 요구사항](../기술-요구사항.md)

---

## 개요

이 문서는 8G 플랫폼의 비기능적 요구사항을 정의합니다. 시스템의 품질 속성과 운영 요구사항을 다룹니다.

---

## NFR-001: 성능

**요구사항**: 시스템은 응답성을 유지해야 한다.

**성능 지표**:

- 페이지 로드: 3초 이내
- 워크플로우 저장: 1초 이내
- 100개 노드 렌더링: 60fps 유지
- 워크플로우 실행 시작: 2초 이내

**검증 방법**:

- Lighthouse 성능 점수 90+ 유지
- React Flow 가상화로 대규모 그래프 지원

**우선순위**: P1 (High)

---

## NFR-002: 확장성

**요구사항**: 시스템은 확장 가능해야 한다.

**확장 포인트**:

- 새 블록 타입 추가: SDK만 수정, UI 자동 생성
- 새 조건 타입 추가: Strategy 패턴
- 워크플로우 수 증가: 데이터베이스 인덱싱

**검증 방법**:

- 1000개 워크플로우 목록 정상 표시
- 새 블록 추가 시 UI 코드 변경 불필요

**우선순위**: P1 (High)

---

## NFR-003: 유지보수성

**요구사항**: 코드는 유지보수 가능해야 한다.

**코드 품질 기준**:

- TypeScript 엄격 모드
- ESLint/Prettier 적용
- 컴포넌트 200줄 이하
- 문서화 원칙 준수

**검증 방법**:

- TypeScript 타입 에러 0개
- Lint 에러 0개
- 200줄 이하 문서 분리

**우선순위**: P1 (High)

---

## NFR-004: 사용성

**요구사항**: 사용자 친화적이어야 한다.

**사용성 기준**:

- 비개발자도 10분 내 워크플로우 생성 가능
- 명확한 에러 메시지
- 실행 중 진행 상황 표시
- 도움말 및 예시 제공

**검증 방법**:

- 사용자 테스트 (5명 이상)
- 평균 학습 시간 측정

**우선순위**: P0 (Critical)

---

## NFR-005: 보안

**요구사항**: 사용자 데이터를 안전하게 보호해야 한다.

**보안 요구사항**:

- 변수에 민감 정보 저장 시 암호화
- SQL Injection 방지 (TypeORM 사용)
- XSS 방지 (React 기본 보호)
- CSRF 토큰 (향후 계획)

**검증 방법**:

- OWASP Top 10 취약점 점검
- 보안 감사 (분기별)

**우선순위**: P1 (High)

---

## NFR-006: 호환성

**요구사항**: 다양한 환경을 지원해야 한다.

**호환성 기준**:

- 브라우저: Chrome 90+, Firefox 88+, Safari 14+
- 운영체제: Windows 10+, macOS 11+, Linux
- 데이터베이스: MySQL 8.0+
- Node.js: 20+

**검증 방법**:

- 크로스 브라우저 테스트
- CI/CD 파이프라인에서 자동 검증

**우선순위**: P1 (High)

---

## NFR-007: 가용성

**요구사항**: 시스템은 안정적으로 동작해야 한다.

**가용성 목표**:

- Uptime: 99.5% (목표)
- 에러 발생 시 자동 재시도
- 부분 장애 시에도 기본 기능 유지

**검증 방법**:

- 로드 테스트 (동시 사용자 100명)
- 장애 복구 시나리오 테스트

**우선순위**: P1 (High)

---

## 기술 제약사항

### TC-001: 브라우저 확장 의존성

**제약사항**: 워크플로우 실행은 8G Extension 설치가 필수이다.

**영향**:

- Extension 미설치 시 실행 불가
- Extension 버전 호환성 관리 필요

**완화 방안**:

- Extension 설치 안내 강화
- 버전 체크 메커니즘 구현

---

### TC-002: 클라이언트 사이드 실행

**제약사항**: 워크플로우는 사용자 브라우저에서 실행된다.

**영향**:

- 서버 리소스 절약
- 사용자 네트워크 환경에 의존
- 스케줄링 불가 (현재)

**완화 방안**:

- 향후 서버 사이드 실행 옵션 추가 계획

---

### TC-003: JSON 컬럼 크기 제한

**제약사항**: MySQL JSON 컬럼 최대 크기 제한이 있다.

**영향**:

- 초대형 워크플로우 저장 제한 가능
- 실행 결과 저장 시 크기 고려 필요

**완화 방안**:

- 워크플로우 분할 권장
- 대용량 데이터는 외부 저장소 사용

---

## 데이터 요구사항

### DR-001: 워크플로우 저장

**요구사항**: 워크플로우는 JSON 형태로 저장되어야 한다.

**데이터 구조**:

```typescript
interface IntegrationAppWorkflowMetadata {
  id: number;
  description?: string;
  meta: FormWorkflow; // JSON 컬럼
  created_at: Date;
  updated_at: Date;
}
```

**검증 방법**:

- Zod 스키마 검증
- 데이터베이스 제약 조건

---

### DR-002: 실행 결과 저장 (향후)

**요구사항**: 워크플로우 실행 결과를 저장할 수 있어야 한다.

**계획**:

- 실행 히스토리 테이블 생성
- 실행 결과 JSON 저장
- 실행 시간, 성공/실패 상태 기록

**우선순위**: P2 (Future)
